## 1. บทนำและวัตถุประสงค์

**FarmSim EDU** เป็นเกมจำลองการเกษตรออนไลน์ เพื่อให้นักเรียนระดับ **มัธยมศึกษาปีที่ 3** ได้เรียนรู้ผ่านการวางแผน ตัดสินใจ และบริหารทรัพยากรฟาร์ม โดยอ้างอิงข้อมูลภูมิภาคจริงด้านภูมิศาสตร์ ภูมิอากาศ และการเกษตร

### เป้าหมายหลัก

- ฝึกการวางแผนระยะยาวและการจัดการทรัพยากร
- เรียนรู้ผลกระทบจากการตัดสินใจในสถานการณ์ที่ไม่แน่นอน
- สนับสนุนการเรียนในห้องเรียนแบบเล่นร่วมกันได้ทันที

### ขอบเขตการใช้งาน

| รายการ | รายละเอียด |
|--------|------------|
| ผู้เล่นต่อห้อง | สูงสุด 8 คน |
| รูปแบบ | ออนไลน์ 100% |
| การเข้าใช้งาน | ไม่ต้องสมัครสมาชิก — ใช้รหัสห้อง + Game PIN + ชื่อ |
| ประเทศที่รองรับ | ประเทศไทย, สหรัฐอเมริกา |

## 2. แนวคิดการออกแบบระบบ

ระบบใช้โมเดล **สองหน้าจอ** แยกบทบาทชัดเจน

| บทบาท | อุปกรณ์ | หน้าที่ |
|--------|---------|--------|
| ครู / ผู้ดำเนินเกม | คอมพิวเตอร์ + จอใหญ่ (Dashboard) | สร้างห้อง แสดงแผนที่ อันดับ เหตุการณ์ |
| นักเรียน / ผู้เล่น | มือถือ (Mobile Web) | วางแผน ปลูกพืช ตอบเหตุการณ์ ทายปัญหา |

> Dashboard ใช้แสดงผลกลางเท่านั้น การตัดสินใจทั้งหมดทำบนมือถือของผู้เล่น

แนวคิดนี้เหมาะกับห้องเรียน: ครูควบคุมจังหวะบนจอใหญ่ นักเรียนมีส่วนร่วมผ่านมือถือของตนเอง

## 3. สถาปัตยกรรมและเทคโนโลยี

### 3.1 ภาพรวมสถาปัตยกรรม

ระบบพัฒนาแบบ **แยก Frontend / Backend** สื่อสารกันผ่าน **REST API (JSON)**  
สถานะเกมเก็บในฐานข้อมูลเป็นศูนย์กลาง และอัปเดตหน้าจอด้วย **Polling** (ไม่ใช้ WebSocket)

```text
[นักเรียน — มือถือ] ──┐
                      ├── Vue.js (Frontend) ── REST/JSON ──► PHP API ──► MySQL
[ครู — Dashboard]   ──┘            ▲                              │
                                   │        Polling สถานะห้อง      │
                                   └──────────────────────────────┘
```

- Frontend ไม่คำนวณผลเกมเอง
- Backend เป็นตัวตัดสินสถานะ คะแนน และเหตุการณ์
- Dashboard และมือถืออ่านสถานะชุดเดียวกัน จึงเห็นผลพร้อมกัน

### 3.2 Frontend

| รายการ | เทคโนโลยี / Framework |
|--------|------------------------|
| Framework หลัก | **Vue.js 3** (Composition API) |
| Build tool | **Vite** |
| Routing | **Vue Router** |
| State management | **Pinia** (จำ session เมื่อเครื่องหลุด) |
| HTTP Client | **Axios** |
| UI Framework | **Bootstrap 5** |
| Animation / Dashboard | **PixiJS** / Sprite sheet |
| อื่น ๆ | QR Code, รองรับ Mobile Web / PWA |

รายละเอียดหน้าจอ การออกแบบ และ Component ที่ใช้ในแต่ละหน้า ดูที่ [หัวข้อ 4](#4-frontend--หน้าจอ-การออกแบบ-และ-component)

### 3.3 Backend

| รายการ | เทคโนโลยี / Framework |
|--------|------------------------|
| Language | **PHP** (Plain PHP — ไม่ใช้ Laravel) |
| API style | **REST API** |
| Routing | Custom Router (`index.php`) |
| โครงสร้าง | MVC แบบเบา: `controllers/` · `models/` · `helpers/` |
| Authentication | ไม่มีระบบ Login — เข้าด้วยรหัสห้อง + Game PIN + ชื่อ |
| Upload | เก็บรูปโปรไฟล์ผู้เล่นบนเซิร์ฟเวอร์ |

### 3.4 Database

| รายการ | รายละเอียด |
|--------|------------|
| DBMS | **MySQL** |
| Charset | `utf8mb4` / `utf8mb4_unicode_ci` (รองรับภาษาไทย) |
| ข้อมูลหลัก | ห้อง, ผู้เล่น, การ์ด, พืช/ภูมิภาค, คะแนน, เหตุการณ์, Hall of Fame |

### 3.5 Real-time

| รายการ | รายละเอียด |
|--------|------------|
| แนวทาง | **Polling** จากฐานข้อมูล |
| เหตุผล | โฮสต์ง่าย ต้นทุนต่ำ ไม่ต้องพึ่งโครงสร้าง WebSocket |

## 4. Frontend — หน้าจอ การออกแบบ และ Component

Frontend แบ่งเป็น **3 กลุ่มหน้าจอ** ตามบทบาทและขนาดจอ

| กลุ่ม | ผู้ใช้ | Layout | จุดประสงค์ |
|-------|--------|--------|------------|
| หน้าเข้าสู่ระบบ | ครู / นักเรียน | เต็มจอ (ไม่บังคับ Layout ร่วม) | เลือกบทบาท สร้าง/เข้าร่วมห้อง |
| Dashboard | ครู (จอใหญ่) | `DashboardLayout` หรือ Lobby แบบเต็มจอ | แสดงสถานะห้อง แผนที่ อันดับ เหตุการณ์ |
| Mobile | นักเรียน (มือถือ) | `MobileLayout` (header เขียว + ความกว้างสูงสุด ~480px) | ตัดสินใจ วางแผน ตอบเหตุการณ์ |

**Layout ร่วม**

| Component | หน้าที่ |
|-----------|---------|
| `DashboardLayout` | กรอบจอใหญ่ — header โลโก้ FarmSim EDU + พื้นหลัง phase; รองรับโหมด fullscreen ตอนจำลองแผนที่ |
| `MobileLayout` | กรอบมือถือ — header sticky สีเขียว + เนื้อหาจัดกลาง ความกว้างจำกัดให้อ่านง่ายบนมือถือ |

**Component ร่วมที่ใช้หลายหน้า**

| Component | หน้าที่ |
|-----------|---------|
| `PlayerAvatar` | แสดงรูปโปรไฟล์ / ตัวอักษรย่อชื่อ |
| `CardSprite` | แสดงการ์ดจาก sprite sheet (ขนาด hand / slot / slot-lg) |
| `CardActionAnimation` | แอนิเมชันตัวละครตามรหัสการ์ด (ปลูก รดน้ำ ฯลฯ) |
| `RegionSelector` | เลือกภูมิภาคของประเทศในห้อง |

---

### 4.1 หน้าแรก (`HomePage`) — `/`

**บทบาท:** จุดเริ่มต้นของทั้งครูและนักเรียน

**การออกแบบ**

- พื้นหลังแบบ phase (บรรยากาศฟาร์ม) จัดกึ่งกลางแนวตั้ง
- โลโก้เป็นจุดโฟกัสหลัก ตามด้วยคำอธิบายสั้น
- ปุ่ม 2 ทางเลือกชัดเจน: **สร้างห้องเกม** (ครู) และ **เข้าร่วมเกม** (นักเรียน)
- แถบสถิติ Hall of Fame เลื่อนระหว่างโลโก้กับปุ่ม

**Component ที่ใช้**

| Component | หน้าที่ |
|-----------|---------|
| `TopScoreMarquee` | ดึง Hall of Fame Top 5 แล้วเลื่อนแสดงชื่อ คะแนน และ `PlayerAvatar` |

---

### 4.2 สร้างห้อง (`CreateRoomPage`) — `/create`

**บทบาท:** ครูเลือกประเทศแล้วเปิด Dashboard Lobby

**การออกแบบ**

- หน้าเต็มจอสไตล์เกม (`create-screen`) — top bar, โลโก้ใหญ่, การ์ดเลือกประเทศ
- การ์ดประเทศ (ไทย / สหรัฐฯ) เป็นภาพคู่พร้อมสถานะเลือก (เครื่องหมาย ✓)
- ปุ่มเริ่มสร้างห้องอยู่ด้านล่าง — สำเร็จแล้วพาไป Lobby Dashboard อัตโนมัติ

**Component / ส่วนประกอบ**

| ส่วน | หน้าที่ |
|------|---------|
| UI ในหน้าเอง | โหลดรายการประเทศจาก API แสดงเป็นปุ่มเลือก ไม่แยก form component เพิ่ม |
| CSS `create-country.css` | สไตล์การ์ดประเทศและปุ่มเริ่มเกม |

### 4.3 Lobby Dashboard (`LobbyDashboardPage`) — `/dashboard/lobby/:roomCode`

**บทบาท:** จอใหญ่ของครูขณะรอผู้เล่นเข้าร่วม

**การออกแบบ**

- Layout แบบ 3 คอลัมน์เต็มจอ (`lobby-screen`) เหมาะกับโปรเจกเตอร์
- Header: โลโก้ · ชื่อห้อง · ปุ่ม **เริ่มเกม** / **ยกเลิกห้อง**
- Status bar: รหัสห้อง · สถานะ · จำนวนผู้เล่น (สูงสุด 8)
- Polling อัปเดตรายชื่อแบบต่อเนื่อง เมื่อเริ่มเกมแล้วพาไป Game Dashboard

**Component ที่ใช้**

| Component | หน้าที่ |
|-----------|---------|
| `LobbyInvitePanel` | แสดงรหัสห้อง, Game PIN, QR Code (ลิงก์ `/join/:roomCode`) และปุ่มคัดลอก/แชร์ |
| `LobbyPlayerGrid` | ช่องผู้เล่น 8 ช่อง — ใช้ `PlayerAvatar` แสดงรูป/ชื่อ/สถานะพร้อม |
| `LobbyGameInfo` | สรุปประเทศ ปีเกม ประเภทฟาร์ม และภูมิภาคที่ผู้เล่นเลือกแล้ว |

### 4.4 Game Dashboard (`GameDashboardPage`) — `/dashboard/game/:roomCode`

**บทบาท:** จอใหญ่ระหว่างวางแผน จำลอง และจบเกม

**การออกแบบ**

- ห่อด้วย `DashboardLayout` — โหมดปกติแสดงสถานะ/อันดับ; โหมด fullscreen ตอนจำลองแผนที่ไทย
- แถบความคืบหน้า 12 เดือนอยู่ด้านบน
- ช่วงวางแผน: แสดงชุดการ์ดตัดสินใจ + ความคืบหน้าว่าใครยืนยันแผนแล้ว
- ช่วงจำลอง: แผนที่เต็มจอ + overlay เหตุการณ์/ควิซ/นำเสนอผล
- ช่วงจบเกม: ตารางอันดับ + รอผู้เล่นส่งผลจากมือถือมาแสดงบนจอใหญ่

**Component ที่ใช้**

| Component | หน้าที่ |
|-----------|---------|
| `DashboardLayout` | กรอบหน้าจอใหญ่ / fullscreen |
| `MonthProgressBar` | แถบ 12 เดือน พร้อมสถานะเดือนปัจจุบันและจับเวลา |
| `ThailandGameMap` | แผนที่ประเทศไทย — วาง marker ผู้เล่นตามภูมิภาค แสดงการ์ด/แอนิเมชันเดือนปัจจุบัน |
| `MapWeatherEffects` | (ภายในแผนที่) เอฟเฟกต์สภาพอากาศตามเหตุการณ์ |
| `CardSprite` | แสดงชุดการ์ดตัดสินใจตอนวางแผน |
| `PlayerJoinList` | รายชื่อผู้เล่น (กรณีไม่ใช้แผนที่ไทย) |
| `BreakingNewsOverlay` | เต็มจอประกาศ Breaking News + นับถอยหลัง |
| `BonusQuizOverlay` | เต็มจอคำถามโบนัส + จำนวนผู้ตอบแล้ว |
| `BonusQuizRevealOverlay` | ประกาศผู้ตอบถูก / ผลควิซ |
| `GameResultPresentation` | นำเสนอสรุปผลผู้เล่นทีละคน (คิวจากมือถือ) พร้อมแผนการ์ดและคะแนน |
| `DashboardAudioControl` | เปิด/ปิดเสียงประกอบ Dashboard |

### 4.5 เข้าร่วมเกม (`JoinPage`) — `/join` และ `/join/:roomCode`

**บทบาท:** นักเรียนกรอกรหัสห้อง + PIN + ชื่อ

**การออกแบบ**

- ใช้ `MobileLayout` — ฟอร์มสั้น อ่านง่ายบนมือถือ
- ถ้าเข้าผ่าน QR (`/join/:roomCode`) จะเติมรหัสห้องให้อัตโนมัติ
- สำเร็จแล้วไปหน้าโปรไฟล์; ถ้ามี session เดิมในห้องเดียวกันอาจข้ามไปหน้า waiting

**Component ที่ใช้**

| Component | หน้าที่ |
|-----------|---------|
| `MobileLayout` | กรอบมือถือ |
| ฟอร์มในหน้า | ช่องรหัสห้อง, PIN, ชื่อ — ไม่แยก component เพิ่ม |

### 4.6 โปรไฟล์ (`ProfilePage`) — `/mobile/profile`

**Component ที่ใช้**

| Component | หน้าที่ |
|-----------|---------|
| `MobileLayout` | กรอบมือถือ |
| UI ในหน้า | preview รูป, ปุ่มกล้อง/แกลเลอรี, ข้าม |

### 4.7 เลือกภูมิภาค (`SelectRegionPage`) — `/mobile/select-region`

**บทบาท:** นักเรียนเลือกพื้นที่ปลูกภายในประเทศของห้อง

**การออกแบบ**

- คำอธิบายสั้นว่าเลือกซ้ำกับผู้อื่นได้
- การ์ดรายการภูมิภาคแบบ 2 คอลัมน์ กดเลือกแล้วยืนยัน

**Component ที่ใช้**

| Component | หน้าที่ |
|-----------|---------|
| `MobileLayout` | กรอบมือถือ |
| `RegionSelector` | โหลดภูมิภาคตาม `country_id` ของห้อง แสดงเป็นปุ่มเลือก |

### 4.8 รอเริ่มเกม (`WaitingPage`) — `/mobile/waiting`

**บทบาท:** หน้ารอหลังตั้งค่าครบ จนกว่าครูจะกดเริ่มเกม

**การออกแบบ**

- แสดง avatar + ชื่อ + ภูมิภาค + รหัสห้อง
- Spinner และข้อความรอ Dashboard
- Polling สถานะห้อง — เมื่อเป็น `planning` / `simulating` / `finished` จะพาไปหน้าถัดไปอัตโนมัติ
- มีปุ่มออกจากห้อง / เข้าห้องใหม่

**Component ที่ใช้**

| Component | หน้าที่ |
|-----------|---------|
| `MobileLayout` | กรอบมือถือ |
| `PlayerAvatar` | แสดงรูปโปรไฟล์ผู้เล่น |

### 4.9 วางแผนการ์ด (`CardPlanPage`) — `/mobile/plan-cards`

**บทบาท:** นักเรียนวางการ์ดตัดสินใจครบ 12 เดือนของปีปัจจุบัน

**การออกแบบ**

## 5. แนวทางการพัฒนาระบบ

### 5.1 แนวทาง Frontend

1. แยกหน้าตามบทบาท: Dashboard (จอใหญ่) และ Mobile (ผู้เล่น)
2. เรียก API ผ่านชั้น `services/` ไม่ยิง Axios ตรงจากหน้าจอ
3. Polling ตามสถานะห้อง (lobby / planning / simulating / finished)
4. แสดงผลจาก state ของห้องเป็นหลัก — การตัดสินใจส่งไป Backend เท่านั้น
5. Build แล้ว deploy แยกจาก API (เช่น Vercel)

### 5.2 แนวทาง Backend

1. ออกแบบ Endpoint ตามโดเมน: ห้อง, ผู้เล่น, Dashboard, Simulation, Hall of Fame
2. ใส่ Logic เกมไว้ใน Model (คะแนน, เหตุการณ์, การ์ด, ตลาด)
3. ใช้ MySQL เป็น Single Source of Truth ของสถานะเกม
4. ใช้ Migration แบบ idempotent (รันซ้ำได้) สำหรับอัปเดต schema และข้อมูลพืช
5. เปิด CORS ให้ Frontend ต่างโดเมนเรียก API ได้

### 5.3 เหตุผลที่เลือกแนวทางนี้

1. **Vue.js + Vite** — พัฒนา UI เร็ว แยกหน้า Dashboard/มือถือได้ชัด
2. **PHP Plain + REST** — โฮสต์ง่าย ต้นทุนต่ำ เหมาะงานการศึกษา
3. **MySQL + Polling** — เล่นพร้อมกันในห้องได้ โดยไม่ต้องโครงสร้าง Real-time ที่ซับซ้อน
4. **แยก Frontend/Backend** — deploy อิสระ อัปเดต UI หรือกฎเกมแยกกันได้

## 6. ขั้นตอนการเล่นหลัก

1. ครูสร้างห้องเกม → เลือกประเทศ (ประเทศไทย / สหรัฐอเมริกา)
2. ระบบแสดงรหัสห้อง, Game PIN และ QR Code บน Dashboard
3. นักเรียนเข้าร่วมด้วยรหัสห้อง + PIN + ชื่อ (+ อัปโหลดรูปโปรไฟล์)
4. นักเรียนเลือกภูมิภาคภายในประเทศของห้อง
5. เมื่อพร้อม ครูกด **「เริ่มเกม」** จาก Dashboard
6. ผู้เล่นวางแผนการ์ด 12 เดือน (ปลูก / รดน้ำ / ใส่ปุ๋ย ฯลฯ)
7. ระบบจำลองทีละเดือน — มี Breaking News, โบนัสทายปัญหา และตลาดกลาง
8. คำนวณคะแนนรายปี → จบเกมแสดงอันดับและบทวิเคราะห์

## 8. วิธีการให้คะแนนและข้อมูลพืชในระบบ

ระบบให้คะแนนจาก **ผลผลิตจริงที่จำลองได้** + **ทรัพยากร/ความสามารถของผู้เล่น** + **ความหลากหลายของแผนการ์ด**  
พืชแต่ละชนิดมีค่าวัดผลตามภูมิภาคและฤดูปลูก ซึ่งส่งผลต่อตัวคูณผลผลิตโดยตรง

> แหล่งข้อมูลหลัก: `ScoreModel.php`, `SimulationModel.php`, `CropPlanModel.php`, `scripts/generate-crops-migration.js`

---

### 8.1 สูตรคะแนนรายปี

```text
คะแนนปี = Production + Resource + Technology + Sustainability
         + Knowledge + Env + Capability
         − floor(Risk / 2)
         − DiversityPenalty
```

| มิติคะแนน | ค่าวัดผล / สูตร | เกณฑ์ |
|-----------|-----------------|--------|
| **Production** | `stock_amount × 2` | ปริมาณผลผลิตในคลังตอนจบปี |
| **Resource** | `floor(coins / 10)` สูงสุด **100** | เหรียญคงเหลือ |
| **Technology** | `tech_level × 15` | ระดับเทคโนโลยี 0–5 (สูงสุด 75) |
| **Sustainability** | ค่า `sustainability` | ความยั่งยืนจากทรัพยากร |
| **Knowledge** | ค่า `knowledge_score` | คะแนนความรู้ (เริ่มต้น 0) |
| **Env** | `max(0, 100 − env_impact)` | ผลกระทบสิ่งแวดล้อมต่ำ = คะแนนสูง |
| **Capability** | `agricultural_capability` | ความสามารถทางการเกษตร 0–100 |
| **Risk** (หัก) | `floor( max(0, 100 − capability) / 2 )` | ความสามารถต่ำ → ความเสี่ยงสูง → หักคะแนน |
| **DiversityPenalty** (หัก) | ตามตารางด้านล่าง | แผนการ์ดซ้ำชนิด / ไม่มีปลูก |

**คะแนนรวมเกม** = ผลรวม `total_score` ทุกปี แล้วจัดอันดับ

#### โทษความหลากหลายของแผน (Diversity Penalty)

| สถานการณ์ | หักคะแนน |
|-----------|----------|
| ไม่มีการ์ดเลย | 80 |
| ชนิดการ์ดไม่ซ้ำ = 1 | 120 |
| ไม่ซ้ำ = 2 | 70 |
| ไม่ซ้ำ = 3 | 35 |
| ไม่ซ้ำ ≤ 5 | 10 |
| ไม่ซ้ำ ≥ 6 | 0 |
| **ไม่มี PLANT** (เพิ่ม) | +40 |

คะแนนปีติดลบได้หากโทษสูงกว่าคะแนนบวก

#### คะแนนโบนัสทายปัญหา (ระหว่างจำลอง)

| ผลตอบ | คะแนน |
|-------|--------|
| ตอบถูกคนแรก | +10 |
| ตอบถูกคนถัดไป | +2 |
| ตอบผิด | −10 |

### 8.2 สูตรผลผลิต (Yield) — เกณฑ์คำนวณ Production

เมื่อเก็บเกี่ยว ระบบคำนวณผลผลิตดังนี้:

```text
yield = round( (50 + floor(soil_quality / 5)) × (capability / 100) × regionMult × seasonMult )
```

| ตัวแปร | ความหมาย | แหล่งค่า |
|--------|----------|----------|
| `soil_quality` | คุณภาพดิน | ทรัพยากรผู้เล่น (เริ่มตามภูมิภาค) |
| `capability` | ความสามารถทางการเกษตร | โปรไฟล์ผู้เล่น (ปรับได้ระหว่างเกม) |
| `regionMult` | ตัวคูณความเหมาะสมภูมิภาค | `crop_region_rates.coin_multiplier` ของพืช×ภูมิภาค |
| `seasonMult` | ตัวคูณฤดูปลูก | ถูกฤดู **1.0** / ผิดฤดู **0.5** |

#### ค่าตัวคูณสำรองเมื่อไม่ตรงเงื่อนไข

| กรณี | ตัวคูณ |
|------|--------|
| พืชถูกภูมิภาค | ใช้ `coin_multiplier` จากฐานข้อมูล (0.25–1.00) |
| พืชผิดภูมิภาค (ไม่มี rate / ไม่เหมาะสม) | **0.35** |
| พืชไม่รู้จักในระบบ | **0.25** |
| ปลูกถูกฤดู | **1.0** |
| ปลูกผิดฤดู | **0.5** |

#### ระดับความเหมาะสมจาก `coin_multiplier`

| ตัวคูณ | ระดับ | ความหมาย |
|--------|-------|----------|
| ≥ 1.00 | excellent | เหมาะสมมาก |
| ≥ 0.85 | good | เหมาะสม |
| ≥ 0.65 | moderate | พอปลูกได้ |
| ≥ 0.45 | poor | ไม่ค่อยเหมาะสม |
| < 0.45 | unsuitable | ไม่เหมาะสม |

> เกณฑ์ “ถูกภูมิภาค” ในระบบ: `coin_multiplier ≥ 0.85`

#### ค่าเศรษฐกิจของพืช

| ค่า | สูตร / เกณฑ์ |
|-----|-------------|
| `base_coin` | ค่าพื้นฐานของพืช (ใช้คำนวณราคา) |
| ราคาซื้อ | `round(base_coin × 0.15)` |
| ราคาขาย | `round(base_coin × 0.30)` |
| `capability_bonus` เมื่อขาย | short **+1** · medium **+2** · long **+3** |
| ระยะเติบโต | `growth_months` (เดือน) — กำหนดเดือนเก็บเกี่ยว |

### 8.6 สรุปเกณฑ์ที่ผู้เล่นควรรู้

1. **เลือกพืชให้ตรงภูมิภาค** → ตัวคูณสูง (ใกล้ 1.00) = ผลผลิตและคะแนน Production สูงขึ้น  
2. **ปลูกในเดือนที่ถูกฤดูของหมวดพืช** → ได้ ×1.0 แทน ×0.5  
3. **ดูแลดินและความสามารถ** → สูตร yield ใช้ทั้ง `soil_quality` และ `capability`  
4. **วางแผนการ์ดให้หลากหลาย** และต้องมีการ์ดปลูก → ลด Diversity Penalty  
5. **ขายผลผลิต / อัปเกรดเทคโนโลยี** → เพิ่ม Resource, Capability และ Technology score

## 11. สรุป
